# PRODUCT PHILOSOPHY

The system is the candidate's most loyal advocate, never a judge or authority. It fights for the candidate using verified evidence. When conflicts exist between candidate claims and documents, the system flags the conflict once, clearly, then follows the candidate's decision without further resistance.

## MENTOR PRINCIPLE

The system is always a mentor first. It never blocks, never gatekeeps, never uses threshold language in user-facing messages.

When quality is low, the system gives honest advice like a trusted friend — "I'm not sure this is right for you" — then immediately offers full support if the candidate wants to proceed.

The mentor philosophy applies to every user-facing message in the system. No rejection language. No threshold language. No authority language. Always honest, always supportive, always ready to fight for the candidate.

**Pre-Beta 1 requirement:** Review every user-facing string in the codebase against this principle before launch. Any message that blocks, gatekeeps, uses threshold language, or positions the system as an authority over the candidate must be rewritten.

**Mentor test — mandatory for every user-facing message before implementation:** Read the message as if you are the candidate. If it sounds like a gatekeeper, a judge, or an authority — rewrite it. The system is always a trusted friend who gives honest advice and then fights for the candidate unconditionally. No threshold language. No rejection language. No authority language. Ever.

# SYSTEM CONSTRAINTS

1. **Domain agnostic** — no finance, legal, medical or any domain-specific logic anywhere in the engine. Test data may be finance but the system must work equally for any profession.

2. **Knockout detection is AI-only** — the requiredProfile layer uses AI to identify knockout criteria from JD language and context. No hardcoded rules.

3. **Output language is candidate-controlled** — candidate selects output language explicitly at application time. System may suggest based on company location but candidate decision is final. Pipeline must support generating output in any language the underlying AI model supports.

4. **Internationally designed** — built for all languages from day one, no hardcoded strings in any language anywhere in prompts or logic.

5. **Candidate is always the final authority over their own profile** — system flags conflicts once, clearly, then follows candidate decision without resistance.

6. **The system is the candidate's most loyal advocate** — never a judge or gatekeeper. It fights for the candidate using verified evidence only.

**Function:** `normalizeStructuredJob()`
**Violation:** Hardcoded list of finance/accounting job title signals used to infer job title when extraction fails (e.g. `buchhalter`, `bilanzbuchhalter`, `controller`, `accountant`, `finance`). Violates SYSTEM CONSTRAINT 1 (domain agnostic).
**Blocked until:** AI track for normalizeStructuredJob is proven stable and rule-based title inference is removed.

## V3 — Hardcoded German strings in `normalizeStructuredJob` and `deriveCompanyContext`
**File:** `lib/orchestration/tailoring/runTailoringPipeline.ts`
**Functions:** `normalizeStructuredJob()`, `deriveCompanyContext()`
**Violation:** Hardcoded German-language fallback labels (`"Unbekannt"`, `"Unbekannte Rolle"`, `"Nicht angegeben"`, `"Finance / Accounting role"`, `"Unknown role"`) used when structured job fields are missing. Violates SYSTEM CONSTRAINT 4 (internationally designed, no hardcoded strings in any language).
**Blocked until:** V1 and V2 are resolved. Fallback labels should be generated by the AI layer or omitted entirely.

# LEGAL COMPLIANCE — TO RESOLVE BEFORE PUBLIC LAUNCH

Items flagged for legal review. **Beta 1 minimum** noted where applicable.

## L1 — Granular consent checkboxes
Users must be able to consent separately to each processing purpose (e.g. pipeline processing, analytics, marketing) rather than a single blanket consent checkbox.
**Beta 1 minimum:** Implement granular consent on signup — at least one checkbox per distinct processing purpose.

## L2 — Right to erasure (Article 17 GDPR)
One-click delete account must permanently erase all personal data across all tables (candidate profile, tailoring runs, any stored CVs or JD text).
**Beta 1 minimum:** Implement delete account option in settings. Full erasure must cascade across all Supabase tables.

## L3 — Right to data portability (Article 20 GDPR)
Users must be able to download all data held about them in a machine-readable format (JSON or CSV).
**Blocked until:** legal review confirms scope and format requirements before public launch.

## L4 — Data retention policy
Define and communicate how long each category of data is retained (tailoring runs, profile data, logs). Implement automated deletion or anonymisation at the defined retention boundary.
**Blocked until:** legal review defines retention periods before public launch.

## L5 — Privacy policy
A privacy policy covering all processing activities must be reviewed and approved by legal before any public-facing launch. Must cover: purposes of processing, legal bases, data transfers, retention, and user rights.
**Blocked until:** legal review and sign-off before public launch.

## L6 — Article 30 processing register
A record of processing activities must be maintained as required by GDPR Article 30. This is an internal compliance document, not user-facing.
**Blocked until:** legal review confirms required entries before public launch.
